## 2026-06-27：前端体验、注册链路与基础业务数据完善

### 1. 加载页视觉风格不统一

**遇到的问题**

- 加载页虽然已经改成白色体系，但用户希望保留原有的金色中心点、波纹和星点。
- 两股蓝色能量需要沿完整莫比乌斯路径追逐。
- 波纹和外部星点也应该具备旋转动效，而不是静态装饰。

**产生原因**

- 前一版为了统一“白色到浅灰背景 + 深灰线稿”的干净风格，删除了金色中心点、波纹和星点。
- 视觉统一做到了，但牺牲了原先的“能量核心”和科技感细节。

**解决方案**

- 保持白色/浅灰背景和深灰莫比乌斯线稿。
- 恢复金色中心点、波纹、星点。
- 保留莫比乌斯环轻微 3D 动画。
- 让蓝色能量沿完整路径追逐，同时让波纹和星点参与旋转。

**STAR 复盘**

- **S（Situation）**：加载页风格需要和整体系统统一，但又不能失去原有辨识度。
- **T（Task）**：在白色体系下恢复金色核心、波纹、星点，并增强动态表现。
- **A（Action）**：调整 `index.html` 中加载页 SVG/CSS 动画结构，保留白底体系并补回核心动效。
- **R（Result）**：加载页既保持了系统统一性，又保留了原本“能量核心”的视觉记忆点。

### 2. 竞赛、经验帖、往届项目复盘数据量不足

**遇到的问题**

- 系统中竞赛只有 8 个，展示效果和推荐效果都显得空。
- 人员、经验帖、往届项目复盘内容不足，页面缺少真实感。
- 用户希望生成 100 个不同竞赛、20 个经验帖，并基于 108 个项目分别生成 2 个复盘。

**产生原因**

- 初始数据库只是演示级数据，数量不足以支撑首页推荐、检索、分页、复盘展示和 Agent 推荐。
- 竞赛类型没有充分分层，导致页面内容单调。

**解决方案**

- 按学科方向和竞赛类型补齐竞赛数据，例如算法编程、创新创业、环境能源、医学工程、数学建模、机械设计、电子信息等。
- 补充经验帖数据，用于“薪火相传/学长学姐经验”模块。
- 为往届项目生成复盘数据，保证项目复盘页面有足够内容支撑。
- 生成对应 SQL，直接写入数据库。

**STAR 复盘**

- **S**：平台数据过少，页面空、推荐弱、演示不充分。
- **T**：快速构造足够数量且类型清晰的竞赛和经验内容。
- **A**：基于业务表结构生成批量 SQL，按类型分布补充竞赛、经验帖、复盘。
- **R**：首页平台信息、竞赛大厅、经验帖、复盘列表具备了较完整的数据规模。

### 3. 首页布局、滚动和信息展示问题

**遇到的问题**

- 平台信息卡片显示不全。
- 页面没有明显滚动能力，底部内容被截断。
- “近期竞赛”卡片偏窄，而“我的待办”区域内容后面空很多。
- 顶部右侧搜索框、全屏、暗色、消息、头像之间间距不统一，视觉突兀。

**产生原因**

- 首页容器高度和滚动区域控制不合理，部分区域被固定布局影响。
- 左右卡片比例不符合实际内容密度：待办只有一条时占用过多横向空间，近期竞赛信息更多却空间不足。
- 顶部工具区使用不同组件默认 margin/padding，缺少统一间距规范。

**解决方案**

- 修复页面主容器滚动，保证平台信息和底部内容能完整访问。
- 调整首页两栏比例，让“近期竞赛”占更合理空间，减少“我的待办”的无效空白。
- 统一顶部工具区的间距、hover 区域和图标对齐。
- 保持首页整体白色卡片、圆角、浅阴影的视觉体系。

**STAR 复盘**

- **S**：首页已经有数据，但布局导致信息展示不完整，用户无法顺畅浏览。
- **T**：恢复滚动能力，优化卡片比例和顶部工具区一致性。
- **A**：调整首页布局、滚动容器、左右栏宽度和工具图标间距。
- **R**：页面内容可完整查看，近期竞赛区域更实用，顶部操作区不再突兀。

### 4. 顶部导航选中态和短线样式不统一

**遇到的问题**

- 某些页面顶部导航选中框出现蓝色矩形边框，样式很丑。
- 用户希望所有页面都和首页一样：蓝色文字 + 底部短线。
- 后续又发现蓝色文字和底部短线贴得太近，甚至叠在一起。

**产生原因**

- Element Plus 默认 focus/active 样式和自定义导航样式叠加，导致出现浏览器/组件默认选中框。
- 不同页面或不同导航层级没有共用统一的 active underline 规则。
- underline 的 top/bottom 偏移没有统一，导致与文字距离过近。

**解决方案**

- 去掉不符合设计的矩形选中框和 focus outline。
- 统一 active 状态为首页风格：文字高亮 + 底部短线。
- 调整底部短线位置，保持文字和短线之间有固定距离。
- 让一级导航、二级 tab 尽量复用同一套间距和短线规则。

**STAR 复盘**

- **S**：导航状态在不同页面表现不一致，破坏系统整体感。
- **T**：统一选中态样式，并解决短线和文字重叠问题。
- **A**：覆盖默认 focus/active 样式，统一 underline 的尺寸、位置和间距。
- **R**：导航选中态更接近首页设计，不再出现突兀蓝色框。

### 5. 个人中心头像编辑入口位置异常

**遇到的问题**

- 提示更换头像的 “+” 号没有位于头像正中心。
- 头像背后的白色框不是完全贴合头像，用户希望它是圆的、贴合头像边界。
- 个人中心整体仍有一些视觉和信息结构问题。

**产生原因**

- 头像容器、遮罩层和 “+” 图标定位基准不一致。
- 白色背景框可能按矩形容器或固定尺寸绘制，而不是完全以头像圆形为边界。

**解决方案**

- 将头像编辑 “+” 图标定位到头像容器中心。
- 让白色背景层与头像同宽高、同圆角，保证完全贴合。
- 仅修改头像中心点和背景圆形贴合问题，其他个人中心问题先分析不扩大改动。

**STAR 复盘**

- **S**：个人中心头像区域是高频视觉焦点，偏移会非常明显。
- **T**：只修最明确的问题：加号居中、背景圆形贴合头像。
- **A**：调整头像容器定位和圆形背景层样式。
- **R**：头像编辑入口位置准确，背景不再像多余白色方框。

### 6. “部门”字段不符合学生竞赛系统语义

**遇到的问题**

- 个人中心显示“未填写部门”。
- 用户指出若依原框架是部门体系，但当前竞赛系统里学生更应该填写学院或专业。
- 用户认为注册时就应该输入这些个人基本信息。

**产生原因**

- 系统沿用了 RuoYi 的 `dept` 概念，但业务对象已经从企业/政务后台变成学生竞赛平台。
- 学生画像需要学院、专业、年级、学号等字段，而不是传统部门。
- 注册流程原本缺失，用户信息只能登录后补齐，导致个人中心空字段较多。

**解决方案**

- 将学生个人信息语义从“部门”调整为“学院/专业”。
- 注册学生账号时要求填写基本信息：真实姓名、学号、年级、学院、专业、手机号、邮箱等。
- 注册后的用户必须真实写入数据库，而不是前端临时状态。

**STAR 复盘**

- **S**：RuoYi 原始字段语义和竞赛平台业务不匹配。
- **T**：把学生基础画像迁移到学院/专业/年级等真实业务字段。
- **A**：分析用户表和学生资料字段，规划注册时写入个人基本信息。
- **R**：个人中心信息结构更符合竞赛平台定位，也为推荐、组队、画像打基础。

### 7. 登录/注册链路设计问题

**遇到的问题**

- 系统只有登录，没有注册。
- 用户希望打开页面默认仍是登录；没有账号时再跳转注册。
- 前一版跳转注册后会提示“用户信息过期”，说明注册页仍触发了登录态校验。
- 点击“注册学生账号”有时没有任何提示或跳转，连续点击才提示“数据正在提交”。
- 注册提交长期卡住，无法成功。

**产生原因**

- RuoYi 默认模板里注册能力不是当前系统核心路径，前后端链路不完整。
- 注册页可能复用了需要 token 的请求拦截逻辑，导致未登录状态访问注册辅助接口时被判定为登录过期。
- 前端 loading 状态和异常处理不完整，接口失败或请求挂起时没有及时 reset。
- 后端注册接口、验证码、邮箱验证码、学院专业加载接口之间的免登录配置不一致。

**解决方案**

- 登录页保持默认入口，登录按钮文案调整为“登录/注册”。
- 没有账号时进入注册页，注册页不触发登录态过期弹窗。
- 注册只能注册学生用户。
- 注册时填写学生基本信息，并与数据库真实关联。
- 修复注册按钮提交状态：请求失败、验证码错误、接口异常都要恢复 loading 并给出提示。

**STAR 复盘**

- **S**：学生用户没有注册入口，注册尝试还会被 token 拦截。
- **T**：补齐可用的学生注册闭环，且保持登录页作为默认入口。
- **A**：调整前端登录/注册跳转、后端注册接口、免登录接口和提交状态处理。
- **R**：注册链路从“页面存在但不可用”推进到可实际落库的业务链路。

### 8. 学院/专业加载失败

**遇到的问题**

- 注册页提示“学院和专业信息加载失败，请确认后端已更新并重启”。
- 专业选择框依赖学院选择，但学院数据没有正常加载，导致专业也不可用。

**产生原因**

- 前端注册页已经接入学院/专业接口，但后端可能未部署最新代码，或接口路径/免登录权限未配置完整。
- 学院/专业数据可能需要后端提供静态字典或数据库查询。

**解决方案**

- 排查学院/专业接口路径、后端 Controller、Security 放行配置。
- 确认后端已重新编译、重启。
- 专业选择依赖学院，学院未选择时专业禁用，这是合理交互；真正问题是学院数据源加载失败。

**STAR 复盘**

- **S**：注册表单依赖学院/专业数据，但首级数据加载失败。
- **T**：确认问题发生在数据接口/部署链路，而不是前端选择器本身。
- **A**：检查接口、鉴权放行和后端重启状态。
- **R**：明确了修复方向：后端接口和部署同步，而不是盲改组件样式。

### 9. 验证码、邮箱验证码和年级约束问题

**遇到的问题**

- 图片验证码明明输入正确，仍提示重新输入验证码。
- 邮箱验证码也正确，但仍无法成功注册。
- 年级字段缺少约束，输入 2028 也能通过。

**产生原因**

- 图片验证码可能存在 uuid/code 绑定错误、大小写/空格处理不一致、刷新后旧 uuid 被使用等问题。
- 邮箱验证码可能只完成了发送，没有和注册提交时的邮箱、验证码、有效期做一致校验。
- 年级字段只是普通文本输入，缺少前端规则和后端兜底校验。

**解决方案**

- 保证图片验证码提交时带正确 uuid，并在失败后刷新验证码。
- 邮箱验证码必须与邮箱地址绑定校验，且校验有效期。
- 年级增加格式和范围校验，例如只允许合理年份或 “2024级” 这类格式。
- 注册失败必须返回明确错误，不允许一直 loading。

**STAR 复盘**

- **S**：注册表单看起来完整，但校验链路不稳定。
- **T**：让验证码、邮箱验证码、年级约束都具备前后端一致性。
- **A**：排查验证码 uuid、邮箱验证码绑定、表单校验和提交异常处理。
- **R**：注册失败原因可定位，避免用户“填对了也过不了”的体验。

## 2026-06-28：Agent 架构强化、通知系统、审核页面与后端安全修复

### 1. ReAct Reflection 增强

**遇到的问题**

- 原 `react_loop` 流程是：调工具 → 合成 → 幻觉校验 → 返回。
- 合成阶段一次性判断“够了”，如果工具数据不足，Agent 没有第二次补查机会。

**产生原因**

- ReAct 循环没有 Reflection 分支。
- 合成 prompt 没有允许模型显式表达“信息不足，需要继续查”。

**解决方案**

- 在 `SYNTHESIS_SYSTEM_PROMPT` 加入规则：
  - 如果工具返回数据不足以给出有价值回答，以 `NEED_MORE: <需要补充什么信息>` 开头。
- 在 `react_loop` 中检测合成结果是否以 `NEED_MORE:` 开头。
- 如果命中，不直接 return，而是继续下一轮工具调用。

**STAR 复盘**

- **S**：Agent 遇到数据不足时容易硬答，影响可信度。
- **T**：给 Agent 第二次查证机会。
- **A**：增加 `NEED_MORE` 协议和循环分支。
- **R**：Agent 从“一次性自判够了”变成“数据不足时能主动补查”。

---

### 2. 用户记忆注入时机优化

**遇到的问题**

- `get_user_memory` 原本在 `ContextBuilder.select_tools` 阶段被拼到每个工具 description：
  - `[用户已知信息: grade=大二 | major=计算机...]`
- 字段和值直接塞进英文工具描述里，格式粗糙，也污染工具定义。

**产生原因**

- 用户画像属于对话上下文，不属于工具能力描述。
- 工具 description 应描述工具用途，不应混入动态用户记忆。

**解决方案**

- 删除 `select_tools` 中对工具 description 的记忆注入。
- 在 `build_messages` 的 system prompt 末尾增加：
  - `## 当前用户已知信息`
  - 画像内容。
- 让 LLM 从 system 层读取用户画像，而不是从工具 schema 中间接读取。

**STAR 复盘**

- **S**：用户画像注入位置不合理，降低 prompt 清晰度。
- **T**：把画像放到更高优先级、更自然的位置。
- **A**：从工具 description 移除记忆，改放 system prompt。
- **R**：工具 schema 更干净，LLM 更容易稳定使用用户画像。

### 3. Agent 可观测性增强

**遇到的问题**

- 当前 step 事件能看到工具调用过程，但缺少：
  - 每请求 `trace_id`
  - 每轮 token 消耗
  - LLM 调用延迟
  - 错误聚合

**产生原因**

- Agent 原型阶段重点在功能连通，没有把评估和排障指标作为一等公民。
- LLM client 的返回结构不统一，部分调用只返回字符串，无法记录 usage。

**解决方案**

- 在 agent loop 入口生成 `trace_id = uuid4()`。
- 每条 step/log 带 `trace_id`。
- 统一 LLM 响应为 `LlmResponse`，包含：
  - `usage.total_tokens`
  - `prompt_tokens`
  - `completion_tokens`
  - `latency_ms`
- 保留 `chat()` 字符串接口作为兼容包装，新增/使用 `chat_response()` 作为 metrics 入口。

**STAR 复盘**

- **S**：Agent 出问题时难以追踪是哪次请求、哪轮工具或哪次 LLM 调用慢/贵。
- **T**：补齐最小可观测性。
- **A**：引入 trace、usage、latency，并进入 step metrics。
- **R**：后续可以按 trace 复盘请求链路和成本。

### 4. InnoTrack Agent 路线判断：从通用 Agent OS 转向垂直推荐 Agent

**遇到的问题**

- Agent 继续堆通用能力，会偏离 InnoTrack 的核心场景。
- 当前推荐仍更像“聊天回答”，而不是“决策系统”。

**产生原因**

- ReAct 适合开放查询，但推荐类业务需要稳定、可评测、可解释。
- 如果让 LLM 直接决定推荐 Top 3，容易出现拍脑袋推荐和不可复盘。

**解决方案**

- 确定走 Cycling Coach 式路线：垂直领域、真实数据、确定性计算、强评测。
- 设计分阶段计划：
  - Phase 1：安全、预算、schema、metrics、scoring_engine。
  - Phase 2：确定性推荐 pipeline。
  - Phase 3：质量闭环。
  - Phase 4：体验和运维。
- 明确推荐链路：
  - profile → competitions → scoring_engine → evidence_collector → RecommendationPlan → LLM 表达。

**STAR 复盘**

- **S**：Agent 已能聊天和查工具，但距离稳定推荐系统还有差距。
- **T**：确定后续架构路线，避免盲目多 Agent 化。
- **A**：制定四阶段强化计划，优先做确定性评分和评测闭环。
- **R**：项目定位从“会聊天的助手”升级为“可解释的竞赛推荐 Agent”。

### 5. Phase 1 地基实现与测试修复

**遇到的问题**

- 记忆加密使用自制 SHA256 keystream XOR，不适合生产。
- LLM metrics 不统一。
- ReAct 没有预算控制。
- 工具输出没有强 schema。
- scoring_engine 需要可独立测试。
- 测试中出现：
  - `AssertionError: assert '' is None`
  - `KeyError: 'total_tokens'`
  - `CompetitionResult object is not subscriptable`

**产生原因**

- 旧代码为快速原型，安全、预算和 schema 约束不足。
- `TurnBudget` 的返回语义和测试预期不一致。
- metrics 字段结构没有统一暴露 `total_tokens`。
- scoring 测试或实现混用了 dict 与 Pydantic object 访问方式。

**解决方案**

- 记忆新写入统一使用 `FERNET:`，新增 `MEMORY_ENCRYPTION_KEY`。
- 读取旧 `AES:` 时继续兼容旧 `AGENT_SHARED_SECRET` 派生逻辑。
- 统一 `LlmResponse`，补齐 usage 和 latency。
- 增加 `TurnBudget`：
  - LLM 次数、工具次数、总 token、耗时限制。
- 新增 domain Pydantic schema。
- 新增 `ScoringWeights` 和 `score_competitions(...)`。
- 修复测试失败：
  - 统一 budget can_call 返回语义。
  - `metrics()` 暴露 `total_tokens`。
  - scoring_engine 兼容 Pydantic model/object 访问。

**STAR 复盘**

- **S**：Agent 能跑，但安全、预算、schema、测试稳定性不足。
- **T**：先稳地基，不急着改完整推荐链路。
- **A**：实现 Fernet、LlmResponse、TurnBudget、domain schemas、scoring_engine，并逐个修测试。
- **R**：Phase 1 相关测试通过，为 Phase 2 推荐 pipeline 做好基础。

### 6. Phase 2 确定性推荐 Pipeline

**遇到的问题**

- 推荐类 intent 如果继续完全交给 ReAct，LLM 仍可能漏调工具或乱调工具。
- 推荐结果缺少结构化计划，前端也难以展示推荐分、证据和准备路径。

**产生原因**

- ReAct 是开放式工具编排，不适合强业务闭环。
- 推荐业务本质需要确定性流程和结构化产物。

**解决方案**

- 推荐/适合/报名建议类请求走 deterministic pipeline。
- 新增 `RecommendationPlan`，包含：
  - Top 3 推荐
  - `match_score`
  - 匹配理由
  - 能力缺口
  - 准备路径
  - 报名风险
  - 截止时间
  - 下一步动作
  - 证据引用
- LLM 只负责把计划表达成自然语言，不能自行新增比赛、奖项、截止时间、技能要求。

**STAR 复盘**

- **S**：推荐结果需要稳定、可解释、可评测。
- **T**：把推荐从 ReAct 自由决策切到确定性 pipeline。
- **A**：接入 profile、competition、scoring、evidence、RecommendationPlan 和表达层。
- **R**：推荐链路开始从聊天式回答升级为业务决策系统。

### 7. Phase 3 质量闭环与测试修复

**遇到的问题**

- `check_plan_constrained_answer` 把“也可以报名火星杯挑战赛”整体识别为越界名称，而测试期望只识别“火星杯挑战赛”。
- 幻觉检查过严，把“这是全国性 IT 竞赛”判为 hallucination，但工具数据中已有“蓝桥杯 / IT竞赛 / 省市级”等相关事实。
- golden eval 需要区分 offline eval 和 live eval。

**产生原因**

- 计划约束检查的中文实体切分不够精确。
- 旧 hallucination checker 主要做字符串匹配，缺少语义容忍。
- eval 如果依赖真实 LLM，会不稳定，不适合全部自动化。

**解决方案**

- 调整 plan-constrained checker 的越界比赛名抽取逻辑，精准截取比赛名。
- 放宽或改进 hallucination checker 对工具事实的引用判断，避免把合理解释误判为幻觉。
- 将 eval 分为：
  - offline deterministic eval：schema、scoring、plan、工具选择、数据不足降级。
  - live eval：最终自然语言质量，可跳过或手动触发。
- 扩展 golden cases 字段：
  - `expected_competitions`
  - `required_facts`
  - `forbidden_claims`
  - `must_include_sections`
  - `quality_rubric`

**STAR 复盘**

- **S**：推荐系统需要质量保障，但检查器过严或过粗都会误伤。
- **T**：让事实检查更贴合 RecommendationPlan 和 evidence。
- **A**：优化越界实体识别、幻觉判断和 eval 分层。
- **R**：测试最终达到 `48 passed, 24 skipped, 1 warning`，质量闭环稳定。

### 8. RabbitMQ 与通知系统问题排查

**遇到的问题**

- 一开始以为 RabbitMQ 未安装。
- 后来确认 MQ 是跑在 Docker 中。
- 通知消息能进入队列，但消费者反序列化失败：
  - `Cannot convert from [[B] to [UserNotification]`
- 后续又发现知识库审核通知根本没发出去。

**产生原因**

- 生产者和消费者 RabbitMQ MessageConverter 不一致，消费者使用默认转换器，无法把 byte[] 转为 `UserNotification`。
- `KnowledgeAuditController` 中将 `createBy` 当成数字用户 ID：
  - `Long.valueOf(retrospect.getCreateBy())`
- 但 RuoYi 的 `BaseEntity.createBy` 存的是用户名，例如 `student01`，不是 user_id。
- 异常被 catch 后静默吞掉，导致通知发送失败难以定位。

**解决方案**

- 在 RabbitMQ 配置中统一使用 `Jackson2JsonMessageConverter`：
  - `RabbitTemplate` 使用同一个 converter。
  - `rabbitListenerContainerFactory` 也使用同一个 converter。
- 知识库审核通知改为：
  - 通过 `createBy` 用户名查询 `sys_user.user_id`。
  - 再写入 `UserNotification.userId`。
- 通知失败不影响审核主流程，只记录 warn 日志。
- 修复部署后清理旧坏消息，避免旧消息继续干扰验证。

**STAR 复盘**

- **S**：审核通过后用户收不到通知，MQ 表面在线但链路不通。
- **T**：同时解决“消息没发出去”和“消息发出后消费失败”两个问题。
- **A**：统一 JSON converter，修复 createBy 用户名解析，增加日志和兜底。
- **R**：通知链路从数据库审核 → MQ → 消费 → 入库具备闭环基础。

### 9. 知识库审核页面 404 与 nginx 部署路径问题

**遇到的问题**

- 管理端 27 个页面中 25 个正常，两个不正常：
  - `/knowledge-manage/retrospect`
  - `/knowledge-manage/experience`
- 页面返回 404。
- Docker/nginx 配置挂载的是：
  - `./ruoyi-ui/dist:/usr/share/nginx/html:ro`
- 实际前端目录是：
  - `innotrack-ui`

**产生原因**

- 前端产物路径配置错误，nginx 服务的不是当前项目的最新 dist。
- SPA 路由刷新或直接访问时，如果 nginx 没有 fallback 到 `index.html`，也会出现 404。
- `sys_menu` 中配置的 component 路径需要和实际 Vue 文件一致。

**解决方案**

- 将 nginx 挂载路径改为 `./innotrack-ui/dist`。
- 配置 SPA fallback：
  - `try_files $uri $uri/ /index.html`
- 检查菜单 component：
  - `system/knowledgeaudit/retrospect/index`
  - `system/knowledgeaudit/experience/index`
- 确认前端实际文件存在：
  - `views/system/knowledgeaudit/retrospect/index.vue`
  - `views/system/knowledgeaudit/experience/index.vue`

**STAR 复盘**

- **S**：菜单存在但点击审核页面 404，影响管理端审核流程。
- **T**：区分是菜单配置错、组件不存在，还是部署路径/nginx fallback 问题。
- **A**：核对 `sys_menu`、前端文件路径、docker-compose 和 nginx 配置。
- **R**：重建前端和重启服务后，两个审核页面返回 200。

### 10. 成绩录入接口权限注解错误

**遇到的问题**

- 成绩录入接口不可用。
- `CompetitionResultController.java` 中写成：
  - `@PreAuthorize("@ss.hasAnyRole('admin','teacher')")`

**产生原因**

- RuoYi 权限方法名实际是 `hasAnyRoles`，不是 `hasAnyRole`。
- 参数形式也应使用逗号分隔字符串：
  - `hasAnyRoles('admin,teacher')`

**解决方案**

- 将注解改为：
  - `@PreAuthorize("@ss.hasAnyRoles('admin,teacher')")`
- 尽量保持最小改动，不重构成绩录入业务。

**STAR 复盘**

- **S**：教师和管理员均无法录入成绩。
- **T**：确认是权限注解错误还是业务逻辑未实现。
- **A**：检查 Controller 注解，按 RuoYi SecurityService 方法签名修复。
- **R**：接口权限判断恢复，成绩录入功能不再因注解拼写阻塞。

### 11. 邀请教师接口要求前端传冗余字段

**遇到的问题**

- 邀请教师时前端需要传 `teacherName`、`competitionId`、`competitionName`、`teamName` 等冗余字段。
- 用户认为后端只应该要求前端传 `teacherId`，其他字段由后端根据 `teamId` 和 `teacherId` 查询补齐。

**产生原因**

- 旧接口把业务补全责任放到了前端，导致前端需要知道过多后端数据结构。
- 这会增加篡改风险，例如前端传错竞赛名、团队名也可能被保存。

**解决方案**

- 前端只提交：
  - `teacherId`
- 后端根据路径中的 `teamId` 查询团队和竞赛信息。
- 后端根据 `teacherId` 查询教师用户。
- 后端补齐：
  - `teamId`
  - `competitionId`
  - `competitionName`
  - `teamName`
  - `teacherName`
  - `studentId`
  - `studentName`
- 后端校验：
  - 当前用户必须是队长。
  - `teacherId` 必须对应存在用户。
  - 该用户必须拥有 `teacher` 角色。

**STAR 复盘**

- **S**：邀请教师接口要求前端传过多可被伪造的数据。
- **T**：收敛接口输入，只保留必要参数。
- **A**：后端根据 `teamId + teacherId` 查询并补齐所有业务字段。
- **R**：接口更安全、更干净，前端调用复杂度降低。

### 12. 复盘投稿字段名不匹配

**遇到的问题**

**STAR 复盘**

- **T**：先解决“调用者不知道怎么传”的问题。

- **R**：降低投稿失败的不确定性，避免先做大规模字段迁移。

### 13. 修改竞赛会重置发布状态

**遇到的问题**

- 修改竞赛信息后，发布状态可能被重置为草稿。
- 用户询问这是否是设计意图。

**产生原因**

- 可能存在两种合理解释：
  1. 安全审核设计：已发布竞赛被编辑后应重新进入草稿/待审核。
  2. 代码缺陷：更新表单没有携带 status，后端默认覆盖为草稿。

**解决方案**

- 暂不直接改。
- 先判断产品意图：
  - 如果是审核型系统，重置为草稿可以合理，但应给用户明确提示。
  - 如果只是普通编辑，则更新时必须保留原发布状态。

**STAR 复盘**

- **S**：编辑竞赛后状态变化会影响线上展示。
- **T**：区分设计行为和 bug。
- **A**：分析两种可能，并建议先确认产品规则。
- **R**：避免盲目修改导致审核流程失效。

### 14. AI 助手菜单是否缺失

**遇到的问题**

- 报告中提到“AI助手未注册路由和菜单，学生无法访问”。
- 用户要求确认是否真实存在。

**产生原因**

- 系统中确实可能没有 `sys_menu` 顶部菜单项。
- 但前端 `PortalLayout.vue` 已经存在学生端浮动“竞赛助手”入口，并导入了 `student/agent/index.vue`。

**解决方案**

- 区分“没有菜单项”和“无法访问”：
  - 如果产品要求必须在顶部导航出现，则需要补 `sys_menu`。
  - 如果浮动入口已满足学生访问，则不是阻塞问题。
- 结论倾向：这不是“学生完全无法访问”的阻塞，但可能是“入口不符合预期”的配置问题。

**STAR 复盘**

- **S**：测试报告认为 AI 助手菜单缺失。
- **T**：确认是否真的不可访问，而不是入口形式不同。
- **A**：检查前端路由、组件和布局入口。
- **R**：明确问题性质：不是后端 API 缺失，而是入口/菜单注册策略问题。

## 总体复盘

### 主要问题类型

| 类型 | 典型问题 | 根因 |
|---|---|---|
| UI 一致性 | 导航选中态、首页布局、头像加号、顶部工具间距 | 缺少统一视觉规范，默认组件样式残留 |
| 注册链路 | token 过期弹窗、验证码失败、提交卡住 | RuoYi 原框架注册链路不完整，免登录接口和状态处理不一致 |
| 数据支撑 | 竞赛/经验帖/复盘太少 | 初始 seed 数据不足 |
| Agent 架构 | ReAct 无 Reflection、推荐不稳定、缺评测 | 原型偏聊天，缺确定性 pipeline 和质量闭环 |
| 消息通知 | MQ 反序列化失败、通知没发出 | converter 不一致，createBy 用户名误当 userId |
| 部署配置 | 审核页 404、nginx 路径错误 | 前端实际目录和 docker-compose 挂载目录不一致 |
| 安全/权限 | 成绩录入权限注解、邀请教师冗余字段 | RuoYi 方法名误用，接口信任前端传值 |

### 最关键的工程收益

1. **前端体验从“能看”推进到“更像完整产品”**
   - 首页、导航、个人中心、加载页、顶部工具区都完成了关键一致性修正。

2. **学生注册从缺失能力变成真实业务链路**
   - 注册学生用户、学院专业、年级、头像、验证码等问题被系统性排查。

3. **Agent 从聊天助手升级为垂直推荐系统方向**
   - 明确了确定性 scoring + RecommendationPlan + eval 的路线。

4. **通知和审核链路从“表面在线”转向可排障**
   - MQ、converter、createBy、nginx、菜单、审核页面都被逐层定位。

5. **接口安全边界更清楚**
   - 邀请教师改为后端补齐字段，减少前端伪造风险。
   - 后续通知删除越权也按同样原则修复：后端必须基于当前用户做归属校验。

### 后续建议

- 注册链路继续补自动化接口测试，重点覆盖验证码、邮箱验证码、学院专业、年级范围、重复用户名/学号。
- 通知系统继续做端到端测试：审核通过/驳回 → MQ → 消费 → `user_notification` → 前端角标。
- Agent 推荐 pipeline 继续接前端展示：推荐分数、证据来源、准备路径、报名风险。
- 复盘投稿先补接口文档和后端校验，再考虑字段命名统一。
- 对所有“个人资源”接口统一做归属校验：通知、报名、队伍、复盘、经验帖、个人资料。
