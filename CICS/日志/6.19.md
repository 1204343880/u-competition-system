# 2026-06-19 开发日志

## 遇到的问题

- 学生端整体仍带有传统后台和 Element Plus 默认风格：页面之间的圆角、阴影、留白、字体层级和动效不统一，个人中心紫色 Banner 与首页割裂，登录页也缺少同一套视觉语言。
- 学生首页偏“系统演示后台”，技术信息和全局统计对学生价值低；四等分统计卡片、均匀快捷入口和厚阴影显得机械，学生登录后还会被权限守卫跳过首页。
- 竞赛大厅卡片过长、信息层级弱、宣传图容易破坏整体简约感；截止时间和操作入口不突出。
- 启动加载页的黑底霓虹莫比乌斯环过于压抑、像 AI 产品初始化，与校园竞赛系统不协调；页面切换也缺少轻量过渡。
- 点击“组队广场”触发 MyBatis 错误：`CompStatus` 枚举被拿去与空字符串比较。
- Agent 抽屉重复出现关闭按钮，工具调用状态重复计数，“理解问题”被误算成信息查询，聊天内容缺少 Markdown 渲染。
- 前端正式构建多次被已有 `dist` 压缩文件占用或 Node 内存限制中断，需要区分代码错误与环境问题。

## 解决思路

- 不强行统一所有页面布局，而是统一“视觉语法”：保留蓝色主色，使用柔和灰白背景、12–16px 圆角、极浅阴影、深灰文字层级和约 200ms 微动效；管理端不改。
- 将学生首页改造成个人工作台：接入报名、队伍、成绩、竞赛和公告等现有接口，突出待办、近期竞赛与快捷入口；统计数字并入不对称 Hero，取消学生登录后跳过首页的逻辑。
- 把宣传图视为受约束的内容而非系统视觉：大厅使用统一小型横版封面，失败或缺图时回退为低饱和分类占位，重新突出标题、主办方、级别、截止时间、状态和详情操作。
- 启动加载采用白底灰线 SVG 莫比乌斯环，资源加载完成后短淡出；路由切换仅做 200ms 淡入和 6px 微上移，避免每次切页都闪现加载层。
- MyBatis 枚举字段只判断 `status != null`，由现有 `CompStatusTypeHandler` 完成枚举与数据库值转换，不再与字符串空值比较。
- Agent UI 只重做展示层，保留 JWT、SSE 和会话逻辑；合并同一次工具调用的运行/完成状态，对模型文本先转义再进行安全 Markdown 渲染。
- 当原 `dist` 被 Windows 占用时改用独立输出目录验证；针对 Node 内存问题采用组件级编译或临时提高构建内存，不修改项目常规配置。

## 原理

- 设计一致性主要来自颜色、排版、间距、圆角、阴影和交互动效等设计令牌，而不是让首页、列表页和个人页采用相同布局。
- 首页应优先回答“我下一步做什么”；个人数据和任务比系统技术栈、全局用户数更符合学生的使用目标。
- 宣传图的视觉不可控，固定比例、裁切方式和占比可以让系统容器保持稳定；完整海报应放在详情页展示。
- SVG 描边动画通过 `stroke-dasharray`、`stroke-dashoffset` 沿路径移动，能以较小成本获得连续线稿动效；启动层放在 Vue 容器外，挂载完成后淡出移除可避免首屏硬切。
- MyBatis OGNL 会真实执行条件表达式；枚举与 `''` 类型不兼容。枚举字段只做空值判断，再交给 TypeHandler 序列化才符合类型边界。
- Markdown 渲染前先转义原始 HTML，可以保留标题、列表、加粗和代码块等格式，同时降低脚本注入风险。
