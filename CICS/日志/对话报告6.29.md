## 一、Task 7 — 成绩录入 SpEL 表达式报错

### 问题描述

前端调用成绩录入接口时，SpEL 表达式报错 `hasAnyRole`（单数），导致接口 500。

### 根因分析

Docker 构建时 Dockerfile 中的 `COPY` 指令指向了旧 JAR 文件名（`ruoyi-admin*.jar`），而实际构建产物为 `innotrack-admin/target/innotrack-admin.jar`。导致 Docker 镜像中打包的是**旧版 JAR**，旧代码使用 `hasAnyRole`（单数），新代码已改为 `hasAnyRoles`（复数）。

### 解决过程

1. **修正 Dockerfile**：`COPY ruoyi-admin*.jar app.jar` → `COPY innotrack-admin/target/innotrack-admin.jar app.jar`
2. **重新构建 JAR**：`mvn clean package -Dmaven.test.skip=true`
3. **重新构建 Docker 镜像**：`docker compose build app`
4. **重启容器**：`docker compose up -d app`
5. **验证**：启动成功，SpEL 表达式不再报错

### 踩坑记录

| 坑 | 原因 | 解决 |
|----|------|------|
| Docker 基础镜像拉取失败 | `eclipse-temurin:17-jre-alpine` 从 Docker Hub 拉取超时/校验失败 | 使用 DaoCloud 镜像源：`docker pull docker.m.daocloud.io/library/eclipse-temurin:17-jre-alpine` 再 `docker tag` 回原名 |
| 旧 nginx 容器名冲突 | 僵尸容器 `ruoyi-nginx` 未清理 | `docker rm -f ruoyi-nginx` |
| `AGENT_SHARED_SECRET` 缺失 | docker-compose.yml 中 app 容器缺少此环境变量，Spring Boot 启动失败 | 从 Agent 端 `.env` 获取密钥值，添加到 `.env` 和 `docker-compose.yml` |
| JVM SIGILL 崩溃 | Spring Boot 启动失败的次生问题 | 修复启动问题后自动消失 |

## 二、Task 8 — Redis / RabbitMQ 连接自动恢复

### 问题描述

- **Redis**：断连后写入永久失败，即使 Redis 恢复也无法自动重连
- **RabbitMQ**：断连后连接永不恢复，监听容器永久停止

### 根因分析

**Redis (Lettuce)**：
- `min-idle: 0` — 空闲连接全部回收，断连后无连接可用，Lettuce 虽有内置重连但无连接触发
- 无 `shutdown-timeout` 配置

**RabbitMQ**：
- `SimpleRabbitListenerContainerFactory` 未设 `missingQueuesFatal=false`，断连后监听容器永久停止
- `RabbitTemplate` 无重试机制，发送消息时连接断开直接抛异常
- 无降级处理，重试耗尽后消息丢失

### 解决方案

#### 2.1 Redis 连接池加固

**文件**：`application.yml`

```yaml
lettuce:
  pool:
    min-idle: 2          # 从 0 → 2，保持最少 2 个活跃连接
    max-idle: 8
    max-active: 16       # 从 8 → 16，应对重连风暴
    max-wait: 3000ms     # 从 -1ms → 3000ms，避免无限阻塞
  shutdown-timeout: 200ms # 新增，优雅关闭超时
```

#### 2.2 RabbitMQ 监听容器容错

**文件**：`RabbitMqConfig.java`

```java
factory.setMissingQueuesFatal(false);  // 队列不可用时不停止监听容器
factory.setConsecutiveActiveTrigger(5);
factory.setConsecutiveIdleTrigger(5);
```

**文件**：`application.yml` / `application-docker.yml`

```yaml
rabbitmq:
  connection-timeout: 10s
  listener:
    simple:
      missing-queues-fatal: false
      concurrency: 1
      max-concurrency: 3
```

#### 2.3 RabbitMQ 发送重试 + DB 降级

**文件**：`RabbitMqProducer.java`

由于 Spring Boot 4.x 不支持旧版 `spring-retry` 的 `RetryTemplate`（API 不兼容），改用**手动指数退避重试**：

```java
// 手动重试（指数退避 1s / 2s / 4s）
for (int attempt = 1; attempt <= MAX_RETRIES; attempt++) {
    try {
        rabbitTemplate.convertAndSend(...);
        return; // 发送成功
    } catch (Exception e) {
        if (attempt < MAX_RETRIES) {
            Thread.sleep(BACKOFF_MS[attempt - 1]);
        }
    }
}
// 重试耗尽后降级：直接持久化到数据库
userNotificationService.insertNotification(notification);
```

### 踩坑记录

| 坑 | 原因 | 解决 |
|----|------|------|
| `spring-retry` 包找不到 | Spring Boot 4.x 不再自动引入 `spring-retry` | 尝试添加依赖 |
| `RetryTemplate` 类型不兼容 | Spring Boot 4.x 使用 `org.springframework.core.retry.RetryTemplate`，与旧 API 不兼容 | 放弃 RetryTemplate，改用手动 for 循环 + Thread.sleep 指数退避 |
| `ExponentialBackOffPolicy` 找不到 | `core.retry` 包中没有这些类 | 最终方案：纯手动实现，不依赖任何 retry 框架 |

### 验证结果

| 测试项 | 结果 |
|--------|------|
| Redis 断连 → 恢复 | `Reconnected to redis` — Lettuce 自动重连成功 |
| RabbitMQ 断连 → 恢复 | 连接 `state: running`，消费者 `active: true` |
| 恢复后永久错误 | 无，ERROR 仅在断连期间出现 |

## 三、Task 9 — 5 项安全缺陷修复

### 问题清单

| # | 问题 | 严重度 |
|---|------|--------|
| 1 | 通知删除归属校验缺失 | 中 |
| 2 | XSS 过滤未覆盖学生端 | 高 |
| 3 | 草稿竞赛可创建队伍 | 中 |
| 4 | SQL 异常信息泄露 | 高 |
| 5 | maxMembers 硬编码为 5 | 低 |

### 3.1 通知删除归属校验

**问题**：`SysNotificationController.remove()` 忽略 `deleteByIds` 返回值，永远返回 HTTP 200。student02 删除 student01 的通知时 SQL 影响 0 行，但前端收到"操作成功"。

**修复**：检查返回值，0 行返回错误。

```java
// 修改前
notificationService.deleteByIds(ids, loginUser.getUserId());
return success();

// 修改后
int deleted = notificationService.deleteByIds(ids, loginUser.getUserId());
if (deleted == 0) {
    return error("通知不存在或无权删除");
}
return success();
```

### 3.2 XSS 过滤覆盖学生端

**问题**：XSS 过滤器 `urlPatterns` 仅覆盖 `/system/*,/monitor/*,/tool/*`，学生端 5 个 Controller（`/student/team`、`/student/hall`、`/student/market`、`/student/agent`、`/student/knowledge`）完全绕过 XSS 过滤。

**修复**：
1. 扩展 `urlPatterns`：`/system/*,/monitor/*,/tool/*,/student/*`
2. `TeamController.createTeam()` 增加防御纵深：`teamName = SafeTextUtils.sanitizePlainText("队名", teamName.trim())`

### 3.3 草稿竞赛组队拦截

**问题**：`CompStatusGuard.requireRegistering()` 混合检查 `publishStatus` 和 `status`，缺少独立的"已发布"守卫。

**修复**：
1. `CompStatusGuard` 新增 `requirePublished()` 方法
2. `CompTeamServiceImpl.createTeam()` 和 `joinTeam()` 增加双重守卫：

```java
compStatusGuard.requirePublished(competition);   // 先校验已发布
compStatusGuard.requireRegistering(competition); // 再校验报名中
```

### 3.4 SQL 异常信息泄露

**问题**：`GlobalExceptionHandler` 的 `RuntimeException` 和 `Exception` 处理器返回 `e.getMessage()`，MyBatis 的 `PersistenceException` 穿透到这些处理器时，泄露完整 SQL 语句、表名、列名。

**修复**：
1. `RuntimeException` / `Exception` handler 改为返回通用消息 `"系统内部错误，请联系管理员"`
2. 新增 `PersistenceException` 专用处理器，返回 `"数据操作失败，请检查输入内容"`
3. `ServiceException` handler 保持不变（业务异常需要返回消息）

### 3.5 maxMembers 动态化

**问题**：`CompTeamServiceImpl.createTeam()` 第 75 行硬编码 `team.setMaxMembers(5)`。

**修复**：从竞赛配置读取，回退默认值 5：

```java
Integer competitionMax = competition.getMaxParticipants();
team.setMaxMembers(competitionMax != null && competitionMax > 0 ? competitionMax : 5);
```

## 五、经验总结

### 部署相关

1. **Docker 镜像源**：国内环境拉取 Docker Hub 镜像不稳定时，使用 DaoCloud 镜像源（`docker.m.daocloud.io`）作为替代
2. **环境变量管理**：新增配置项时同步检查 `.env`、`docker-compose.yml`、`application-*.yml` 三处
3. **JAR 产物命名**：多模块项目中 Dockerfile 的 COPY 路径必须与实际构建产物一致

### 中间件相关

4. **Redis 连接池**：`min-idle` 不能为 0，否则断连后无连接触发重连。建议至少设为 2
5. **RabbitMQ 监听容错**：`missingQueuesFatal` 必须设为 `false`，否则断连后监听容器永久停止
6. **Spring Boot 4.x 兼容性**：`spring-retry` 旧 API 不兼容，建议手动实现重试逻辑而非引入框架

### 安全相关

7. **删除操作返回值**：Controller 必须检查 Service 层的返回值，不能永远返回成功
8. **XSS 过滤范围**：`urlPatterns` 必须覆盖所有业务路径，不能只覆盖框架默认路径
9. **异常信息脱敏**：`RuntimeException` / `Exception` 的全局处理器绝不能返回 `e.getMessage()`，防止内部信息泄露
10. **防御纵深**：关键校验（如权限、状态）应在多个层次重复校验，不依赖单一检查点
