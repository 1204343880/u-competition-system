### 🚀 竞赛系统“大厂级”技术栈武装地图

你可以把项目分为“一期（基础业务跑通）”和“二期（架构升维抗压）”。以下是二期你可以名正言顺引入的技术：

**1. 高并发秒杀场景（引入 Redis 缓存 + MQ 消息队列）**

* **业务痛点：** 像“蓝桥杯”或“互联网+”这种全校爆款竞赛，报名通道开启的瞬间，几万学生同时疯狂点击“立即报名”。如果直接写 MySQL，数据库瞬间就会被打挂。
* **技术武装：** 引入 **Redis** 预热竞赛名额（库存预扣减），拦截 99% 的无效请求；引入 **RabbitMQ / RocketMQ**，将报名成功的请求扔进队列异步排队落库（削峰填谷）。

**2. 分布式并发安全（引入 Redisson 分布式锁）**

* **业务痛点：** 在“自由组队大厅”，一个热门“大腿”队伍只差 1 个人就满员了。此时 3 个同学在同一毫秒点击了“申请加入”。普通的 Java `synchronized` 锁不住多台服务器的并发，会导致队伍超编。
* **技术武装：** 引入 **Redis 分布式锁（Redisson）**。在代码里加锁，确保同一时刻只有一个请求能操作这个队伍的成员数据，完美解决“超卖”问题。

**3. 海量数据的高性能检索（引入 Elasticsearch全文搜索引擎）**

* **业务痛点：** 你的“薪火相传”模块未来如果沉淀了十万篇历届学长的长篇经验贴、路演 PPT 提取文本。学生想要搜索包含“机器狗视觉识别”的帖子，MySQL 的 `LIKE '%...%'` 会导致全表扫描，卡死整个系统。
* **技术武装：** 引入 **Elasticsearch (ES)**。将经验帖数据同步到 ES 中，实现类似百度的毫秒级分词、高亮全文检索，瞬间拉满系统的高级感。

**4. 复杂关系与推荐算法（回归你的 Python 规划）**

* **业务痛点：** “Skill 双向匹配引擎”如果只靠 SQL 查标签，太死板了。
* **技术武装：** 利用你规划的 **Python + FastAPI 微服务**，甚至引入图数据库（Neo4j），计算学生能力标签与队伍需求的“余弦相似度”，实现智能推荐。

### ⚠️ 魔鬼架构师的现实警告

虽然一个项目可以装下所有，但**饭要一口一口吃，路要一步一步走。**

如果你连最基础的 MySQL 表结构都没建好，连 Spring Boot 的增删改查都还没写顺，现在就去搞 Redis 和 MQ，系统一定会变成一个“运行不起来的怪物”**。大厂面试官不光看你用了什么，更看重你**为什么用**以及**落地的细节。

我们现在的首要任务，是为你未来的“高性能双向匹配”打下最坚实的数据地基——也就是我们上一回合停滞的地方：**改造 `sys_user` 表，扩展学生的专属信息。**

既然你有了宏大的架构愿景，那我们就用最严谨的态度对待当前这一步。请向我下达指令：**你准备先去 Navicat（数据库）里写 `ALTER TABLE` 语句新增“学号 (student_no)”等字段，还是直接在 Vue 前端的个人中心页面开始画输入框？**
